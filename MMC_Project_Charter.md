## 1. 项目概述

### 1.1 项目背景

随着个人智能设备数量不断增加（手机、平板、笔记本、智能家居等），用户面临以下痛点：

- **品牌生态割裂**：跨品牌设备之间难以互通（如华为手机无法方便地传文件给小米笔记本）
- **操作繁琐**：需要在多个 App / 系统界面之间切换操作
- **传输受限**：文件传输依赖微信/邮件/云盘，受限于文件大小与网速
- **体验缺失**：缺乏统一、轻量、隐私友好的多设备管理体验

### 1.2 项目简介

**MMC（Multi-device Mesh Control）** 是一款局域网内跨品牌设备的发现、配对与文件传输工具。

**核心定位**：让一台手机控制身边所有智能设备，在不同设备之间自由传输文件。

**核心价值主张**：
- 打开即用：无需注册登录，无需繁琐设置
- 隐私优先：数据仅在局域网内传输，不上云
- 跨平台：一套代码覆盖 Android / iOS，后续扩展 PC

## 2. 项目目标与范围

### 2.1 总体目标

| 目标 | 描述 | 验收标准 |
|------|------|----------|
| 设备发现 | 30秒内完成两台同局域网设备的发现与配对 | 计时器测量 ≤30s |
| 文件传输 | 传输速度满足日常使用需求 | ≥10 MB/s（千兆局域网） |
| 屏幕镜像 | 实时显示目标设备屏幕 | 延迟 ≤200ms |
| 冷启动 | 应用快速启动 | ≤3秒 |
| 远程控制 | Android端可远程操控其他设备 | 功能可用 |

### 2.2 MVP 范围

**纳入 MVP**：
- Android ↔ Android 文件传输（单文件 + 批量）
- mDNS 设备发现与配对
- TLS 1.3 加密传输
- 断点续传
- 配对历史管理
- 核心库 Rust 实现

**不纳入 MVP（后续迭代）**：
- 跨 VLAN / 互联网传输（NAT 穿透）
- 屏幕镜像（延迟优化版）
- 远程触控
- iOS 客户端
- PC 客户端

### 2.3 项目边界

| 边界 | 说明 |
|------|------|
| 适用网络 | 纯局域网（同一 Wi-Fi 或网段），不支持互联网中转 |
| 适用平台 | Android 8.0+（MVP），iOS 14.0+、PC（后续） |
| 用户群体 | 个人用户，无需企业功能 |
| 数据范围 | 仅传输用户主动选择的文件，不采集任何用户数据 |

### 3.2 核心功能需求

| 功能模块 | 关键需求 |
|----------|----------|
| **设备发现** | 基于 mDNS/Bonjour 自动发现同局域网设备，≤3秒完成 |
| **设备配对** | ECDH 密钥交换 + TLS 1.3 加密，≤30秒完成 |
| **文件传输** | TCP 直连 + 分片传输，≥10MB/s，支持断点续传 |
| **屏幕镜像** | H.264 硬编码，≤200ms 延迟（MVP 后） |
| **远程控制** | 触控事件 WebSocket 实时传输（MVP 后） |

### 3.3 非功能需求

| 指标 | 目标值 | 来源需求 |
|------|--------|----------|
| 冷启动时间 | ≤3秒 | IR-017 |
| 内存占用 | ≤100MB（空闲） | SR-060 |
| 传输速度 | ≥10 MB/s | IR-011 |
| 镜像延迟 | ≤200ms | IR-015 |
| 设备发现 | ≤30秒（含配对） | IR-018 |

## 4. 技术方案概要

### 4.1 技术栈选型

| 层级 | 技术选型 | 说明 |
|------|----------|------|
| 核心共享库 | **Rust** | 内存安全、高性能、跨平台编译成熟 |
| Android | **Kotlin + Jetpack Compose** | 现代安卓开发标准，冷启动性能好 |
| iOS | **Swift + SwiftUI** | 平台 API 深度集成（后续阶段） |
| PC | **Tauri** | 与 Rust 核心库无缝 FFI（后续阶段） |
| 跨语言桥接 | **UniFFI** | 自动生成 Kotlin/Swift 绑定 |
| 序列化 | **Protocol Buffers v3** | 跨平台一致 |
| 网络发现 | **mDNS/DNS-SD** | Android NsdManager / iOS NetService |
| 加密 | **TLS 1.3 + ECDH Curve25519** | rustls 实现 |
| 视频编码 | **H.264 硬编码** | Android MediaCodec |
| 文件校验 | **BLAKE3** | Rust 原生，高性能 |

### 4.2 系统架构

```
┌──────────────────────────────────────────────────────────────────┐
│                        表现层 (UI)                                │
│   [Android: Kotlin + Compose]    [iOS: Swift + SwiftUI]           │
│   [PC: Tauri + WebView]                                           │
└───────────────────┬──────────────────────────────────────────────┘
                    │  UniFFI / 原生 Bridge
                    ▼
┌──────────────────────────────────────────────────────────────────┐
│                MMC Core Library (Rust)                            │
│                                                                  │
│  ┌─────────────┐  ┌──────────────┐  ┌─────────────┐            │
│  │ Discovery   │  │ Pairing      │  │ File Transfer│            │
│  │ (mDNS)      │  │ (ECDH+TLS)   │  │ (TCP+chunk)  │            │
│  └─────────────┘  └──────────────┘  └─────────────┘            │
│  ┌─────────────┐  ┌──────────────┐  ┌─────────────┐            │
│  │ Screen      │  │ Remote Ctrl  │  │ Security    │            │
│  │ Mirror      │  │ (Event Stream)│  │ (rustls)    │            │
│  └─────────────┘  └──────────────┘  └─────────────┘            │
│                                                                  │
│  Common: tokio async runtime, prost, BLAKE3                      │
└───────────────────┬──────────────────────────────────────────────┘
                    │
                    ▼
┌──────────────────────────────────────────────────────────────────┐
│                   平台抽象层 (Platform Abstraction)                │
│   [Android: NsdManager, MediaProjection, SAF, KeyChain]          │
│   [iOS: NetService, ScreenCaptureKit, FileProvider, Keychain]    │
└──────────────────────────────────────────────────────────────────┘
```

### 4.3 通信协议

| 信道 | 协议 | 用途 |
|------|------|------|
| 控制信道 | WebSocket over TLS 1.3 | 配对指令、触控事件、元数据 |
| 数据信道 | TCP 直连 + TLS | 大文件传输 |
| 心跳 | 自定义协议（5s 间隔） | 设备在线状态检测 |

## 8. 风险评估

| 风险 | 概率 | 影响 | 等级 | 应对措施 |
|------|------|------|------|----------|
| 路由器禁用 mDNS 多播 | 中 | 高 | 中 | 提供手动 IP 输入降级方案 |
| Android 后台传输被系统杀死 | 中 | 中 | 中 | Foreground Service + 网络锁 + 用户引导 |
| 屏幕镜像延迟不稳定 | 中 | 中 | 中 | 码率自适应 + 丢帧策略 |
| 触控事件隐私限制 | 中 | 中 | 中 | 设计为"协作式"，需双方确认 |
| Rust 团队学习曲线 | 低-中 | 低 | 低-中 | 提供 UniFFI 脚手架降低门槛 |
| iOS 屏幕采集授权弹窗体验差 | 低 | 低 | 低 | 清晰权限引导 + 降级说明 |

## 9. 可行性结论

| 维度 | 结论 |
|------|------|
| **技术可行性** | ✅ 基本可行。核心功能（P2P发现、加密传输、文件互传）技术成熟，无根本性障碍。屏幕镜像和远程控制有挑战但可解决。 |
| **产品可行性** | ✅ 可行。差异化明确（品牌无关+隐私优先+远控），市场需求真实。 |
| **经济可行性** | ✅ 可行。零服务器成本，适合小团队或独立开发者。 |
| **法律合规性** | ✅ 可行。隐私友好（不上云、不注册），合规风险低。 |
| **综合结论** | ✅ **建议启动项目**，优先实现文件传输核心功能，屏幕镜像和远程控制作为后续迭代。 |
