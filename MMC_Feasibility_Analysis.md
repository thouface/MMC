# MMC — 可行性分析报告

---

## 1. 技术可行性

### 1.1 设备发现与配对

| 技术点 | 可行性 | 分析 |
|--------|--------|------|
| mDNS/DNS-SD 局域网发现 | ✅ 可行 | iOS native Bonjour、Android NsdManager 均成熟稳定。跨平台 Rust crate 有 `mdns-sd`。 |
| Diffie-Hellman 密钥交换 + TLS 1.3 | ✅ 可行 | `rustls` + `x25519-dalek` 在 Rust 生态中稳定。Android (Conscrypt) 和 iOS (SecureTransport) 均支持 TLS 1.3。 |
| 30 秒内完成发现+配对 | ⚠️ 有条件 | 在理想网络环境下可行。若设备在不同 VLAN、或路由器禁用多播，需增加手动 IP 输入降级方案。 |
| 无需账号注册 | ✅ 可行 | 纯设备证书 + 配对确认即可，无需用户体系。 |

**技术风险**：部分路由器默认关闭 mDNS 多播（企业级路由器常见），导致发现失败。需设计手动发现降级方案。

### 1.2 文件传输

| 技术点 | 可行性 | 分析 |
|--------|--------|------|
| ≥10 MB/s（千兆局域网） | ✅ 可行 | TCP 直连 + 分片传输，理论上限约 100+ MB/s。实测主要受限于磁盘 IO。 |
| 断点续传 | ✅ 可行 | 基于 chunk 索引 + BLAKE3 校验，业界成熟方案。 |
| 批量传输 | ✅ 可行 | Tokio async 多路复用，5 路并发。 |
| Android 后台传输 | ⚠️ 有条件 | Android 8+ 对后台服务限制严格。需 Foreground Service + 网络锁，可能导致用户感知通知栏常驻。 |
| 跨品牌传输 | ✅ 可行 | P2P + ProtoBuf 与品牌无关。 |

**技术风险**：Android 低端设备磁盘写入速度可能成为瓶颈（eMMC 约 50-100 MB/s），不会影响传输速度目标，但文件解压/保存可能较慢。

### 1.3 屏幕镜像

| 技术点 | 可行性 | 分析 |
|--------|--------|------|
| MediaProjection 屏幕采集 | ✅ 可行 | Android 5+ 支持，需用户授权。 |
| H.264 硬编码 | ✅ 可行 | Android MediaCodec / iOS VideoToolbox 硬编，帧率 30fps 可达。 |
| ≤200ms 延迟 | ⚠️ 有挑战 | 端到端路径：采集→编码→传输→解码→渲染。理论最低延迟约 100-150ms，需精细优化（如零拷贝、硬件加速）。 |
| 音视频同步 | ✅ 可行 | 基于 PTS 时间戳同步，偏差 ≤50ms 可实现。 |

**技术风险**：200ms 延迟目标在高质量网络下可达，但若 Wi-Fi 信号弱或丢包率高，延迟会显著上升。需设计码率自适应降级策略。

### 1.4 远程控制

| 技术点 | 可行性 | 分析 |
|--------|--------|------|
| Android 触控事件捕获 | ⚠️ 受限 | Android 10+ 对后台触控事件有限制，`AccessibilityService` 可行但需用户开启辅助功能。 |
| 坐标映射准确性 | ⚠️ 有难度 | 不同分辨率设备间映射需动态校准，边缘触控误差较大。 |
| 键盘映射 | ✅ 可行 | Android 软键盘事件可通过 InputConnection 转发。 |
| 剪贴板同步 | ✅ 可行 | Android 10+ ClipboardManager API 受限，但可通过 `ContentProvider` 或无障碍服务方案变通。 |

**技术风险**：Android 隐私限制导致后台触控事件难以可靠获取。替代方案：作为"被控端"镜像时在通知栏常驻快捷操作。

### 1.5 冷启动性能

| 技术点 | 可行性 | 分析 |
|--------|--------|------|
| ≤3 秒冷启动 | ✅ 可行 | Kotlin Coroutine + Lazy 加载 + Rust Core 预加载，中端机（Snapdragon 665+）可达成。 |
| ≤100MB 内存占用 | ⚠️ 有条件 | Rust 核心库约 10-20MB，Kotlin 运行 + Compose UI 约 50-80MB。需严格控制图片/资源加载。 |

**技术风险**：Compose UI 在低端机冷启动较慢，可考虑 `Jetpack App Startup` 优化初始化顺序。

## 2. 产品与市场可行性

| 维度 | 分析 | 结论 |
|------|------|------|
| **市场需求** | 跨设备文件传输是强需求，现有方案（微信、AirDrop、局域网快传）均有痛点：微信有大小限制且隐私问题，AirDrop 仅限 Apple，局域网快传功能单一 | ✅ 存在差异化空间 |
| **差异化壁垒** | 品牌无关 + 隐私优先 + 远控 是独特组合。AirDrop 只解决 Apple 内部，LocalSend 功能较单一 | ✅ 竞争壁垒明确 |
| **商业模式** | MVP 阶段不考虑变现。后续可探索：高级功能订阅、设备管理云服务（可选）、企业版 | ⚠️ 商业模式需后续探索 |
| **政策风险** | 不上云、纯局域网，符合隐私监管趋势（GDPR、个人信息保护法）。无需版号、内容审核 | ✅ 政策友好 |

## 3. 经济可行性（研发成本估算）

| 阶段 | 工作量估算 | 说明 |
|------|------------|------|
| **核心库 (Rust)** | 4-6 人月 | mDNS、加密、配对、文件传输、屏幕流、远程控制 |
| **Android App** | 3-4 人月 | UI、平台集成、权限处理 |
| **iOS App** | 2-3 人月 | SwiftUI + 平台 API |
| **PC 客户端** | 3-4 人月 | Tauri + WebView |
| **测试 + 调优** | 2-3 人月 | 兼容性、性能、安全测试 |
| **MVP 总计** | **14-20 人月** | 团队 3-4 人，约 4-6 个月 |

| 成本项 | 估算 |
|--------|------|
| 服务器/云服务 | 0（纯 P2P，无需服务器） |
| 开发工具 | 0（均为开源/免费） |
| 证书签名 | Apple Developer ($99/年)、Google Play ($25/次) |
| 人员成本 | 视团队情况 |

**结论**：MVP 阶段无需服务器，适合小团队或独立开发者。

## 4. 法律与合规可行性

| 维度 | 分析 | 结论 |
|------|------|------|
| **隐私合规** | 不采集用户数据、不上云、不注册账号，符合 GDPR、个人信息保护法 | ✅ 合规 |
| **网络合规** | 纯局域网传输，不涉及跨境数据传输 | ✅ 合规 |
| **权限合规** | Android 权限：网络、存储、屏幕采集、辅助功能；均为主流 App 常见权限 | ✅ 合规 |
| **专利风险** | mDNS、Wi-Fi Direct、TLS 等均为公开标准协议，无专利壁垒 | ✅ 低风险 |

## 5. 风险汇总

| 风险 | 等级 | 应对措施 |
|------|------|----------|
| 路由器禁用 mDNS 多播 | 中 | 提供手动 IP 输入降级方案 |
| Android 后台传输被系统杀死 | 中 | Foreground Service + 网络锁 + 用户白名单引导 |
| 屏幕镜像延迟不稳定 | 中 | 码率自适应 + 丢帧策略 + 延迟/画质动态权衡 UI |
| 触控事件隐私限制 | 中 | 设计为"协作式"而非"监控式"，需双方确认 |
| Rust 团队学习曲线 | 低-中 | 提供完整的 UniFFI 绑定脚手架，降低各端开发门槛 |
| iOS 屏幕采集授权弹窗体验差 | 低 | 清晰的权限引导 + 降级说明 |

---

## 6. 可行性结论

| 维度 | 结论 |
|------|------|
| **技术可行性** | ✅ **基本可行**，核心功能（P2P发现、加密传输、文件互传）技术成熟，无根本性障碍。屏幕镜像和远程控制有挑战但可解决。 |
| **产品可行性** | ✅ **可行**，差异化明确，市场需求真实。 |
| **经济可行性** | ✅ **可行**，零服务器成本，适合小团队。 |
| **法律合规性** | ✅ **可行**，隐私友好，合规风险低。 |
| **综合结论** | ✅ **建议启动 MVP**，优先实现文件传输核心功能，屏幕镜像和远程控制作为后续迭代。 |
